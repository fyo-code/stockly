## The Project

Building a demand forecasting engine for **Mobexpert**, one of Romania's largest furniture retailers. The business problem: Mobexpert buyers currently order inventory based on gut feel and basic spreadsheets. The goal is to predict how many units of each SKU will sell in the next 4–8 weeks so buyers can make data-driven stocking decisions.

**The client situation:** Mobexpert is the target, not yet a confirmed paying client. Building toward a demo/pilot. One store's data is available (Baneasa, Bucharest). The engine is being built on real Mobexpert transaction data.

**The product scope is broader** — dead stock detection, supplier reliability scoring, scenario simulation — but all of that is deprioritized. The forecasting engine is the core and everything effort is going there.

## What Was Built — The Forecast Engine

**Location:** `backend/forecast_engine/`

**Architecture:** 7 forecasting methods run in parallel on all A-tier SKUs. Their outputs are aggregated via plain median. The median prediction per SKU is the final forecast.

### The 7 Methods

| Method | File | What it does |
|---|---|---|
| ETS | `methods/ets_model.py` | Exponential smoothing — fits trend + seasonality to weekly history |
| Anomaly-Adjusted | `methods/anomaly_adjusted.py` | 13-week rolling baseline with spike detection |
| Multi-Scale Lag | `methods/multi_scale_lag.py` | Weighted average of 4w/8w/12w/26w lagged demand |
| Calendar Events | `methods/calendar_events.py` | Seasonal multipliers based on calendar month patterns |
| Category-Relative | `methods/category_relative.py` | SKU demand as a fraction of category total, scaled by category forecast |
| Naive Seasonal | `methods/naive_seasonal.py` | Same period last year (52-week prior) |
| Crostons | `methods/crostons.py` | Intermittent demand method — separate smoothing for demand intervals and sizes |
| LightGBM | `methods/lgbm_model.py` | Gradient boosting — **ARCHIVED** (disabled via ENABLED_METHODS flag) |

**Key pipeline files:**
- `backtesting.py` — core runner: `load_a_tier_weekly()`, `run_all_methods()`, `score_predictions()`, `save_predictions()`
- `backtest_windows.py` — windowed backtester (6 historical windows for hypothesis testing)
- `aggregation.py` — `EnsembleForecaster` class, plain median + weighted variants
- `recency_filter.py` — drops SKUs silent for 10+ weeks before forecasting, re-attaches as explicit zeros
- `seasonal_dampener.py` — applies seasonal multiplier to ensemble output for SKUs with high monthly CV
- `bias_correction.py` — applies per-method multiplicative correction before ensemble
- `derive_biases.py` — derives bias values from backtesting results
- `category_routing.py` — per-category method routing (currently empty — routing rejected in testing)
- `config.py` — `ENABLED_METHODS` toggle, `RECENCY_FILTER_WEEKS`, `SEASONAL_DAMPENING_ENABLED`
- `ingestion_mobexpert.py` — CSV → SQLite ingestion pipeline

**Bias correction values:** `active_docs/iter4_b6_biases.json`
- Derived from W1–W5 historical windows
- Applied as `p_corrected = p / (1 + bias_capped)` where cap is ±0.30
- ETS: −23%, Multi-Scale Lag: −23%, Calendar Events: −23%, Naive Seasonal: −23%, Anomaly-Adjusted: −21%, Category-Relative: +27%, Crostons: +43%

## The Four Iterations — What Happened

### Iteration 1 (baseline test)
- **Scope:** 20 high-revenue SKUs, Nov–Dec 2024 prediction
- **Result:** Aggregate was close (+3% on total units) but individual SKU accuracy was poor
- **WMAPE:** 56.6%, hit±20%: 39%
- **Finding:** Small sample, can't trust; scale up

### Iteration 2
- **Scope:** Full A-tier (6,358 SKUs), Jan–Feb 2025 prediction
- **Result:** Catastrophic
- **WMAPE:** ~120%+, 4,217 of 6,358 predicted SKUs sold zero (phantom demand everywhere)
- **Root causes identified:**
  - LightGBM was wildly over-predicting
  - Anomaly-Adjusted was routing to wrong SKU types
  - No recency filter — engine predicted demand for SKUs silent for months
  - No Naive Seasonal method
  - No Croston's for intermittent demand

### Iteration 3 (real improvement)
- **Scope:** Full A-tier, Mar–Apr 2025 prediction
- **Changes made:** 5 targeted fixes:
  1. LightGBM archived
  2. Anomaly-Adjusted restricted to SMOOTH SKUs only
  3. Naive Seasonal added (same-period-last-year)
  4. Seasonal dampening added (CV-based multiplier)
  5. Recency filter added (silence SKUs with 0 sales in last 10 weeks)
- **Real results (Mar–Apr 2025 actuals):**
  - WMAPE: 84.9%
  - **Hit ±20%: 21.6%** ← best individual SKU accuracy achieved so far
  - Bias: +57.3% (systematic over-prediction)
  - Beats naive baseline

### Iteration 4 (algorithmic improvement phase + live prediction)

**The algorithmic work (not a real iteration — no new actuals):**

Introduced windowed backtesting — 6 historical train/predict windows (W1: May-Jun 2024 through W6: Mar-Apr 2025) to test hypotheses properly. Every hypothesis had to improve ≥4/6 windows to be accepted.

Hypotheses tested:

| Hypothesis | Result | Reason |
|---|---|---|
| B.1a: Archive ETS + Multi-Scale Lag | REJECTED 6/6 windows | Shrinking method pool hurt median stability |
| B.1b: Archive Anomaly-Adjusted + Category-Relative | REJECTED 4/6 windows | Same root cause |
| B.2: Weighted ensemble (3 variants) | REJECTED all 3 | Conservative methods upweighted → median biased low |
| B.3: ACCESORII category routing | REJECTED 4/6 windows | Removing methods from ACCESORII removed corrective votes |
| **B.6: Per-method bias correction** | **ACCEPTED 4/6 windows** | Fixed systematic over-prediction |

**Bias correction (B.6):** derived each method's mean signed error from W1–W5, capped at ±0.30, applied multiplicatively. Mean WMAPE improved 78.1% → 75.2% across 6 historical windows.

Also done in Iter 4:
- Phase C: Parsed 609 Baneasa discount rows → `promo_calendar` + `sku_promo_weeks` tables
- Phase D: Promo lift analysis — MOBILIER DE CASA lift=1.104 (CI too wide for modeling), all others insufficient data
- Phase E: Scoring report on windowed backtests

**Bug found and fixed:** Recency filter had a sparse-data bug — `weekly_demand` only stores rows with sales, so SKUs silent for 10+ weeks have no rows in the recent window. Filter was finding zero silent SKUs (checking for `net_sold <= 0` among rows that don't exist). Fixed with left-merge against all training SKUs. Post-fix: 4,327 SKUs correctly silenced.

## The Pivot — What We Figured Out

At the end of this session the user made an explicit strategic decision:

> **Individual SKU accuracy (hit ±20%) is the only metric that matters. WMAPE is secondary. Every decision from here must be evaluated first by its impact on individual SKU hit rate.**

The reasoning: buyers make decisions at the SKU level. Aggregate WMAPE can look great while the engine is useless for actual purchasing decisions. The money is in correctly predicting the high movers — the 134–155 SKUs that sell 6+ units/month represent 44–50% of actual unit volume.

**Current best individual SKU accuracy: 21.6% (Iter 3). Every 5th selling SKU is predicted within ±20%.**

---

## Current Problem Statement

The engine is architecturally wrong for individual SKU accuracy. Here's why:

**1. Wrong methods for the problem.** ETS, Naive Seasonal, Multi-Scale Lag — these are time-series methods designed for continuous regular demand. Furniture demand is discrete and intermittent. The correct tool is a feature-based count regression model (LightGBM with Tweedie distribution), which was archived in Iter 1.

**2. Wrong aggregation strategy.** Median of 7 mediocre predictions has a hard accuracy ceiling. For individual SKU precision, you need the best model for each demand regime — not the average of many models.

**3. Using one store only.** Training on Baneasa alone gives almost no signal for SKUs that sell 1–2 units/month there. With 4 stores, the same SKU might show 4–8 units/month of signal — enough to learn a real pattern.

**4. Uniform bias correction.** One bias factor per method applied to all SKUs destroyed high-volume SKU accuracy. High-volume SKUs needed zero or upward correction; the uniform correction pushed them down 44–74%.

**5. Wrong objective in optimization.** Every backtesting decision was made on WMAPE. WMAPE and hit±20% are different objectives. Optimizing one can and did hurt the other.

**6. ETS is catastrophically wrong.** Live prediction showed ETS at 145.2% WMAPE with +94.4% bias — predicting nearly double actual. It looked mid-pack in backtesting (82.2% WMAPE) because its worst behavior only manifested at the May-Jun inflection point. The bias correction cap (+30%) was nowhere near enough to absorb the actual ~120%+ raw over-prediction. ETS needs to be archived.

---

## What Is Achievable — Honest Assessment

**For full A-tier (including 1–2 unit sellers):** ceiling is ~35–45% hit±20% with the best possible approach. Cannot go higher — 49% of selling SKUs sold exactly 1 unit; predicting "0.8–1.2 exactly" for hundreds of intermittent items is not statistically feasible.

**For high movers (6+ units/month):** 60–75% hit±20% is achievable. These SKUs have real, learnable patterns. This is what major furniture retailers report with proper ML. This is where the business value is.

**The realistic near-term target:** 35–40% overall hit±20%, 60%+ on high movers.

## Recommended Iter 5 Plan (in priority order)

### Step 1 — Multi-store ingestion
Ingest data from all 4 stores into the DB. This is the single biggest unlock — it multiplies the training signal per SKU by ~4x without changing any methods. A SKU with 1 sale/month in Baneasa becomes a SKU with 4 sales/month chain-wide.

### Step 2 — Archive ETS immediately
Evidence is conclusive from live data. ETS is destabilizing the ensemble. Disable via `ENABLED_METHODS["ets"] = False` in `config.py`. No regression risk — every backtesting test showed ETS was mid-pack by coincidence, not because it's good.

### Step 3 — Demand-regime routing with separate model pools
Split A-tier SKUs into 3 tiers based on historical monthly velocity:
- **High movers (5+ units/month):** run LightGBM Tweedie + Calendar Events + Multi-Scale Lag
- **Medium (2–4 units/month):** Crostons + Naive Seasonal + Calendar Events
- **Intermittent (≤1 unit/month with recent sales):** binary classifier (will it sell?) + predict 1 if yes
- **Dormant:** recency filter handles this already

### Step 4 — LightGBM Tweedie rebuild
The code exists in `methods/lgbm_model.py`. It was archived because it underperformed on WMAPE — but WMAPE is no longer the primary metric. Re-enable it for high-mover SKUs only, with features:
- Own 4w/8w/12w/26w/52w demand lags
- Category total demand
- Month-of-year seasonality dummies
- Price tier (if available)
- Cross-store demand (if multi-store data ingested)
- Demand pattern classification (SMOOTH/LUMPY/INTERMITTENT/ERRATIC from abc_tiers)

### Step 5 — Volume-tier bias correction
Replace the uniform per-method bias correction with per-volume-tier correction:
- Tier L (<5 units/month): current correction was roughly appropriate
- Tier H (5+ units/month): derive separate bias from high-mover SKUs only — likely zero or upward correction needed
This directly addresses the regression from Iter 3 to Iter 4 on high-volume SKUs.

### Step 6 — Evaluate everything on SKU hit rate first
All future hypothesis testing via the windowed backtester must report hit±20% by volume bucket as the primary metric, not WMAPE. A change is accepted only if it improves hit±20% on high-mover SKUs across ≥4/6 windows.

## The North Star

**Individual SKU hit±20% on high-mover SKUs. Everything else is secondary.**

The argument: if you can correctly predict how many units of the top-selling 150 SKUs will move in the next 4 weeks, you've covered ~50% of all unit volume with high accuracy. That directly drives buying decisions. Nailing the 1-unit intermittent sellers is nice to have but not where the commercial value is.

The current approach (median ensemble of 7 time-series methods, optimized for WMAPE) has been shown to have a hard ceiling of around 20–25% overall hit±20%. Hitting 60%+ on high movers requires a feature-based ML model (LightGBM Tweedie), multi-store training data, and demand-regime routing.

That's the pivot. That's Iter 5.
